# Changelog

## Unreleased

### Management capabilities тАФ Corrective Actions (CAPA) and OSH Committee
- Prioritized ahead of Noise Management: before the next technical/document module, built the cross-cutting management capabilities requested тАФ a Corrective Action (CAPA) register, action tracking, OSH Committee management, and a genuine Management Dashboard.
- New `Actions` table is the single engine behind both "CAPA" and "Action Tracking" тАФ presented in the UI as one register (`/capa`, `public/js/pages/actions.module.js`) branded "Corrective Actions (CAPA)": the default quick filter is Corrective+Preventive (CAPA), with "All Actions" as one more filter chip in the same register rather than a second module. Fields include Date Raised, Assigned Department, and Representation (added while the schema was still small, strengthening future reporting) alongside Action Type/Priority/Status/Assigned To/Due Date/Root Cause/Measures/Effectiveness Review/Evidence. Optional Source Module + Source Reference trace an action back to whatever finding raised it (a failed inspection, a CHRA finding, a committee decision) тАФ same free-text-pointer convention as `Activity Log`'s `Record Reference`, not a hard link, since the source can be any table in the app.
- New OSH Committee module (`/osh-committee`, master = `OSH Committee Meetings`) covering meeting schedule, agenda and minutes, plus a separate `OSH Committee Members` roster page (`/osh-committee/members`) тАФ kept as its own small dedicated page rather than a sub-table, since members and meetings are a many-to-many relationship the module framework's master/sub-table shape doesn't fit. Reuses the existing `openRecordForm` modal for add/edit, zero new form code.
- The top-level Dashboard is now a genuine cross-module Management Dashboard: added a Chemical Management KPI row (previously Machinery was the only module represented тАФ itself a pre-existing gap), a Corrective Actions/CAPA KPI row (Open/Overdue/CAPA Open), an OSH Committee snippet (active members, next scheduled meeting), and an "Open Corrective Actions" table listing the 5 most overdue actions across every source module.
- Two small, generic, backward-compatible additions to the module framework (`public/js/framework/moduleFramework.js`), both proven out by these two modules and available to any future one: `config.relatedLinks` (extra quick-action buttons to a related-but-not-sub-table page) and `extraOverviewKpis` now also receiving the master record list (previously only sub-table data), since Corrective Actions/CAPA has no sub-tables to compute its Open/Overdue KPIs from otherwise. Both modules also confirmed the framework degrades cleanly to zero sub-tables (`subTables: {}`) тАФ the Dashboard/Register/Profile/Documents/Activity shape still works with nothing hanging off the master record.
- New Airtable tables: `Actions`, `OSH Committee Meetings`, `OSH Committee Members`. Verified end to end: live Airtable round-trips (including an overdue-action `isOverdue` computation check), and a full headless-browser pass over the Management Dashboard, the CAPA register (confirmed the CAPA quick filter is selected by default), an Action profile, the OSH Committee dashboard, and the Committee Members page, with zero console errors.

### Platform Foundation тАФ Event Bus, Configuration Service, AI Extraction Logging
- Added `server/lib/eventBus.js`, a thin wrapper around Node's `EventEmitter`, as the platform's standard inter-module communication mechanism going forward. `buildModuleRouter` (`server/lib/moduleRouter.js`) gained an optional `eventPrefix` config field тАФ when set, create/update/delete/upload emit `<Prefix>.Created/.Updated/.Deleted/.Uploaded`; every attachment upload also emits a generic `Document.Uploaded` regardless of module. Wired up for the whole Chemical Management route family (`Chemical`, `Substance`, `SDS`, `StorageInspection`, `LabelInspection`, `Waste`, `Training`, `ExposureMonitoring`) plus explicit `SDS.RevisionCreated`/`SDS.RevisionSuperseded` events in `sdsDocuments.js`'s bespoke supersede logic. Every other module (Machinery, HRA, HIRARC, SOP) omits `eventPrefix` and is byte-for-byte unaffected. No consumers exist yet тАФ this only prepares the platform for a future Notification Engine, Workflow Engine, Audit Trail, n8n, or AI Agents to subscribe.
- Added `server/lib/config.js`, a centralized Configuration/Feature-Flag service backed by `server/config/features.json`, plus `GET /api/config/flags` so the frontend reads the same flags. File-backed for now (all flags default to enabled, so nothing existing changes); the interface is what a future Airtable- or Admin-panel-backed store would implement behind, same abstraction pattern `airtable.js` already uses for the data layer.
- Added AI Extraction Logging: a new `AI Extraction Logs` Airtable table, `server/lib/aiExtractionLog.js`, and instrumentation in `server/lib/extract.js` around both `extractLicenseData` and `extractSDSData` (model, prompt version, duration, success/failure, warnings) тАФ purely observational, never affects the extraction's own return value or thrown errors. Also emits `AIExtraction.Completed`/`AIExtraction.Failed` on the Event Bus. Feeds a future AI Operations Center; no consumer built yet.

### OSH-C Portal v2.0 тАФ Chemical Lifecycle Management redesign
- **Substances (new module):** every SDS Section 3 (Composition/Information on Ingredients) hazardous ingredient is now automatically extracted and stored as its own linked record on a new `Substances` table (Chemical Name, CAS/EC/REACH Number, Concentration, Hazard Classification, Signal Word, H/P Statements, GHS Pictograms) тАФ one Product, many Substances. `server/lib/extract.js`'s SDS extraction prompt/schema gained a `substances[]` array (same "null if not visible, never invent" contract); the "+ Add Chemical" wizard bulk-creates them after saving the chemical and its first SDS revision, and shows a preview table in the review step.
- **Product Name vs. Chemical Name:** `Chemicals`' name field is relabeled "Product Name" (field ID unchanged тАФ the trade/product identity for the register); the SDS-defined "Chemical Name" concept now lives per-ingredient on `Substances` instead of being conflated with the product's trade name. `Internal Code` relabeled "Product Code". No data moved or was lost тАФ both are display-name-only Airtable relabels.
- **SDS expiry, automatically calculated:** `SDS Documents` gained an `Expiry Date` Airtable formula field (`Revision Date + 5 years`). A new time-based Current/Expiring Soon/Expired status is computed server-side/client-side from it (90-day "Expiring Soon" window) тАФ distinct from the existing `Status` field's Current/Superseded revision-history semantics, which is unchanged.
- **Storage, merged:** the Chemical Profile's Storage tab now shows the chemical's storage profile (Cabinet, Maximum/Current Quantity, Storage Method, Temperature, Ventilation, Segregation, Incompatible Chemicals тАФ 7 new additive `Chemicals` fields) directly above its Storage Inspection history, via a new generic `profileHeaderFields` primitive on `moduleFramework.js`'s sub-table profile tabs. `Chemical Storage Inspection` gained a `Corrective Action` field.
- **Documents, centralized:** the Chemical Profile's separate Photos and Attachments tabs are replaced by one aggregated, read-only Documents tab collecting every uploaded file across the chemical and all its linked sub-tables (SDS files, inspection photos, training certificates, waste disposal certificates, CHRA reports, тАж) by reference тАФ nothing is re-uploaded or duplicated. New generic `documents: { mode: "aggregated" }` primitive on `moduleFramework.js`; `chemicals.js`'s profile route computes the flattened list server-side via a new `postProcess` hook on `buildProfileRoute` (`server/lib/profileAggregation.js`) тАФ fully additive, every other module's profile route is unaffected.
- **General Information, enriched:** now shows Product Code, Current/Maximum Quantity, and a derived block тАФ Current SDS, its time-based Expiry Status, Latest Revision, Manufacturer, Physical Form (all pulled from the current SDS Documents record, never duplicated onto Chemicals) тАФ plus a Compliance Summary (SDS Available, CHRA Completed, and the four conditional-module flags below, Next Review from the soonest linked CHRA review date). New generic `generalInfoDerived` hook on `moduleFramework.js`.
- **Label Inspection / Waste Management, expanded:** Label Inspection gained Label Condition and Corrective Action fields (its existing attachment field is now used as a multi-file Photos gallery). Waste Management gained Scheduled Waste Code and Disposal Certificate fields, and three relabels: Waste Type тЖТ Waste Category, Contractor тЖТ Licensed Contractor, Manifest тЖТ Consignment Note.
- **Conditional modules:** four new assessor-set (not AI-inferred, a deliberate compliance-auditability decision) `Chemicals` fields тАФ Exposure Monitoring / LEV / Biological Monitoring / Health Surveillance Required тАФ gate their respective Profile tabs, grouped together after CHRA. Exposure Monitoring already has a real sub-table; LEV/Biological Monitoring/Health Surveillance don't exist yet (future milestones), so their tabs show a short placeholder instead of a broken list тАФ the module is future-ready without another redesign, per the brief.
- **Register, redesigned:** Filter panel (Hazard Classification/Storage Location/Supplier/Manufacturer/Physical Form/SDS Status), free-text search, Quick Filters (All/Hazardous/Non-Hazardous/Current SDS/Expiring Soon/Expired SDS), Export Excel (client-side CSV, no new dependency) and Export PDF (browser print, same pattern as the existing DOSH register), and a "ЁЯУД View SDS" column that opens the latest revision directly тАФ fetched fresh at click time, never a cached Airtable attachment URL, since those are short-lived. Powered by a new `GET /chemicals/reports/register-data` aggregation endpoint and a new generic `registerFilters` primitive on `moduleFramework.js`'s Register page, reusable by future modules.
- **Every KPI card is now clickable:** `Components.kpiCard` gained `tabKey` (jumps to that sub-table's tab within the same Module Dashboard, via a new delegated `[data-kpi-tab]` listener) and `navigate` (routes to a different page, e.g. the master count тЖТ the Register). Applied generically to every module-framework KPI, plus a new "Expired SDS" KPI specific to Chemical Management (time-based, doesn't fit the generic per-sub-table shape) via a new `extraOverviewKpis` hook.
- **Master-Detail Cockpit (new, additional interface):** `/chemical/cockpit` тАФ left pane is the Chemical Register (reusing the register-data endpoint and search), right pane is a read-only, collapsible Overview (new `Components.mountAccordion`, sibling to the existing `mountTabs`) covering every Profile section, lazily rendered per section on first expand, updating with no page reload as a different chemical is selected. An "Open Full Chemical Profile" button links to the existing, unmodified Chemical Profile page тАФ this does not replace it, per the brief.
- New Airtable tables: `Substances`, `AI Extraction Logs`. All other schema changes are additive fields or display-name-only relabels (field IDs unchanged) тАФ every existing field, record, route, and URL keeps working exactly as before. Verified end to end: real Airtable MCP schema creation, live API round-trips against production data (register-data, profile aggregation, compliance summary, document aggregation), and a full headless-browser pass over Dashboard тЖТ Register (filters/search/export) тЖТ Profile (all 10 tabs) тЖТ Master-Detail Cockpit (list, selection, accordion expand) with zero console errors.

### Add Chemical wizard refactor тАФ drag & drop, extraction retry, AI-field highlighting, new workplace fields
- Rebuilt Step 1 of the "+ Add Chemical" wizard as a proper drag-and-drop PDF-only dropzone (plus Browse) тАФ upload is mandatory before the form appears, and non-PDF files are rejected client-side with a clear message.
- Added an explicit extraction-progress state (spinner + status text) between upload and the review form, and a failure state with **Retry Extraction** and **Continue Manually** actions тАФ extraction failure no longer silently drops the user straight into a blank form.
- AI-populated fields on the review screen are now visually highlighted (accent border + "AI" badge) so it's clear at a glance what came from the SDS vs. what needs the user's own input тАФ matches the existing convention from Machinery's license-extract flow.
- Reorganized the wizard's workplace section around the fields actually needed day to day тАФ Storage Location, Department, Responsible Person, Quantity, Unit, Internal Code, Review Frequency, Status тАФ with the DOSH-Register-specific fields (Process, Workers Exposed, Type of Use, Control Measures, PPE Actually Used, Internal Remarks) kept in a clearly separated "Additional Details" section rather than removed, since the generated DOSH register depends on them.
- Added 5 new additive `Chemicals` fields: Responsible Person, Unit, Internal Code, Review Frequency, Status. Every existing field and record is unchanged.
- The first SDS for a new chemical is now explicitly recorded as Revision 1 when the AI doesn't extract a version number, instead of being saved with a blank version.
- **Found and fixed a real latent bug** while wiring the new fields: the previous milestone added `Department`/`Process`/`Control Measures`/`PPE Actually Used`/`Type of Use`/`Internal Remarks` to the *backend* schema (`server/lib/schema.js`) but never to the *frontend* `MODULES.chemicals.fields` map тАФ so `saveWizard()`'s field lookups (`CF["Department"]`, etc.) were silently resolving to `undefined` and writing to a bogus key that Airtable would reject or ignore. Fixed by completing the frontend field map; verified by resolving every field key the wizard uses against the live config and confirming each maps to a real Airtable field ID, plus a live create round-trip.
- No extraction logic was duplicated тАФ the wizard still calls the same `server/lib/extract.js` тЖТ `extractSDSData()` тЖТ `/api/chemicals/extract-sds-preview` pipeline built in the previous milestone; only the upload UI and review-form UI around it changed. SDS versioning, the supersede-on-new-Current rule, and audit logging are all unchanged and unaffected by this refactor.

### DOSH-compliant Chemical Management тАФ SDS-driven intake, versioned SDS Library, Training, generated Chemical Register
- Rebuilt "+ Add Chemical" as an SDS-driven wizard: upload the Safety Data Sheet, AI reads it (new `extractSDSData` in `server/lib/extract.js`, same Gemini pattern as Machinery's existing license extraction), and one combined review screen shows every extracted field pre-filled and editable alongside blank workplace-specific fields (storage location, department, process, workers exposed, quantity, control measures, PPE actually used, type of use, remarks). Saving creates the Chemical record and its first SDS Documents record together, linked.
- Added a real, versioned SDS Library (`SDS Documents` table) тАФ SDS revisions are never overwritten or deleted; saving a new revision as the current one automatically supersedes the chemical's previous current revision server-side, preserving full document history. Supports multiple languages per chemical (English / Bahasa Malaysia / Other) and tracks whether a revision was AI-extracted, for future extraction-quality tracking.
- Added a sixth Chemical Management sub-table, `Chemical Safety Training`, purely as configuration on the module framework тАФ no new page-rendering code, proving the framework's "new operational area = config, not code" value again.
- Added a generated DOSH Chemical Register (`/chemical/dosh-register`): reproduces the DOSH "Guidelines for the Preparation of a Chemical Register" (USECHH Regulations 2000) Section A (Company Information)/Section B (per-chemical hazard-and-usage table)/Section C (Prepared By/Reviewed By) layout entirely from data already in the system тАФ nothing is filled in twice. Two report columns (CSDS, Label compliance) are computed at generation time rather than stored. Print-optimized for the browser's native "Print to PDF" тАФ no server-side PDF-generation dependency added. Every generation is logged to a new `DOSH Register Generations` audit table.
- New Airtable tables: `SDS Documents`, `Chemical Safety Training`, `Company Settings`, `DOSH Register Generations`. New additive fields on `Chemicals`: Department, Process, No. of Workers Exposed, Control Measures, PPE Actually Used, Type of Use, Internal Remarks тАФ every existing field and record untouched.
- Added one new framework primitive, `customAddHandler`, so a module can swap in a bespoke "+ Add" flow instead of the default record-form overlay (`public/js/framework/moduleFramework.js`) тАФ additive and backward-compatible; Machinery keeps its existing default behavior unchanged.
- Found and fixed a real design gap during implementation: the generic `buildModuleRouter` factory's default "add a custom route after the factory call" pattern (used everywhere else in the app) can't intercept-before-create, which the SDS supersede rule needs. `server/routes/sdsDocuments.js` composes a small custom router in front of the generic one instead, so the business rule lives at the data layer and is correct regardless of which UI path creates a record.
- Verified live end to end: workplace fields on Chemicals, SDS supersede-on-create (old revision confirmed still retrievable, not deleted), profile aggregation including the two new sub-tables, DOSH register data aggregation (including a real CSDS=Y / Label=тАФ derivation check), Company Settings create/edit round-trip, generation audit logging, and non-admin permission enforcement on every new route. Live Gemini extraction accuracy could not be verified in this environment (no configured `GEMINI_API_KEY`) тАФ confirmed instead that the failure path degrades gracefully (clear error, not a crash) both server- and client-side.

### Business-function dashboard redesign тАФ Machinery module
- Reorganized the platform's navigation around business functions instead of raw document tables: the Dashboard now shows Machinery, Chemical Management, Noise Management and Operational Safety as full workspaces, following `Dashboard -> Module Dashboard -> Register -> Profile -> History/Documents/Reports`.
- Added a client-side History-API router (`public/js/router.js`), a reusable UI component kit (KPI cards, data tables, tabs, breadcrumb, activity feed тАФ `public/js/components/`), and a service layer (`public/js/services/`) separating data access from rendering. No build step or framework introduced тАФ still plain HTML/CSS/vanilla JS, same Render deployment.
- Rebuilt the Machinery module end to end: a Machinery Dashboard (Overview, Machine Register, Certificates of Fitness, Preventive Maintenance, Corrective Maintenance, Inspection, Calibration, Risk Assessment, Documents, Reports) and a full Machine Profile page per machine (General Information, Maintenance History, CF Certificate, Inspection History, Repair History, Photos, Attachments, Risk Assessment, Activity Timeline).
- Added 5 new Airtable tables тАФ Machinery CF, Preventive Maintenance, Corrective Maintenance, Machinery Inspection, and a shared Calibration Records table (designed to be reused by the future Noise Management module) тАФ plus a `Photos` field on Machinery. All link back to Machinery by record, avoiding duplicated data.
- Added a `GET /api/machinery/:id/profile` endpoint that aggregates a machine with every linked sub-record in one call, and new CRUD routes for each new table, built on the existing `buildModuleRouter` factory (no new backend pattern needed).
- Noise Management and Operational Safety are not yet rebuilt тАФ their Dashboard tiles currently open the same classic record list/form HRA, HIRARC and SOP have always used, so nothing already working was removed. Full target schema and page design for both are documented in DATABASE.md and PROJECT_ROADMAP.md ahead of their own milestones.
- Fixed a data-correctness bug found during verification: Airtable's `filterByFormula` resolves a linked-record field to the linked record's primary-field text, not its record ID, so a naive `FIND(machineId, ARRAYJOIN({Machine}))` filter would have silently returned zero results. The profile endpoint instead fetches each sub-table and filters on the actual linked record ID in application code.

### Generic module framework + Chemical Management module
- Extracted the Machinery Dashboard/Register/Profile pages, sub-record CRUD form, and profile-aggregation endpoint into a declarative config-driven engine: `public/js/framework/moduleFramework.js` (frontend тАФ `defineBusinessModule(config)`) and `server/lib/profileAggregation.js` (backend тАФ `buildProfileRoute(config)`). A module is now a config object (master table, linked sub-tables, optional reused risk-assessment table, attachments, tab layout) instead of hand-written pages per module.
- Refactored Machinery onto the framework (`public/js/pages/machinery.module.js` replaces the old `machineryDashboard.js`/`machineryRegister.js`/`machineryProfile.js`/`machineryService.js`) and re-verified it end to end тАФ live CRUD on all 5 sub-tables, linked-record aggregation, file uploads, admin/non-admin permissions, route specificity, and tab-state across saves тАФ before building anything on top of it, per the explicit "Machinery must show zero regression first" requirement. Also hardened the router itself: route matching now prefers static routes (`/machinery/register`) over dynamic ones (`/machinery/:id`) regardless of `<script>` tag order, removing a latent fragility.
- Built Chemical Management directly on the framework as proof it generalizes: `public/js/pages/chemical.module.js` is the only new page-rendering-adjacent code (a small config file plus one bespoke tab for Emergency Spill Procedure) тАФ no new Dashboard/Register/Profile logic was written. Chemical Management Dashboard, Chemical Register, and a Chemical Profile page per chemical, covering Exposure Monitoring, Storage Inspection, Label Inspection, Waste Management, SDS Library, CHRA (reused as Risk Assessment), Photos, Attachments and Activity Timeline.
- New Airtable tables: `Chemicals` (master/inventory), `Exposure Monitoring`, `Chemical Storage Inspection`, `Chemical Label Inspection`, `Waste Management`; a `Chemical` linked-record field added to the existing `CHRA` table (its old free-text substance field is untouched); a `Document Type` field added to `SOP` (SOP / Work Instruction / Spill Response) so Emergency Spill Procedure тАФ and the future Operational Safety Work Instructions tab тАФ filter the existing table instead of duplicating it.
- The Chemical Management tile on the main Dashboard now opens the real module instead of the classic CHRA record list; CHRA itself is retired as a standalone destination and lives on as Chemical Management's Risk Assessment tab, matching the original "CHRA should no longer exist as a top-level module" requirement.
